Basic Tutorial

The basic tutorial creates a pipeline that splits data to two branches and merges it before writing to the destination. You'll use data prevew to help configure the pipeline, and you'll create a data alert before starting the pipeline.

Here are high level steps we'll take to build the basic pipeline:
  1. Configure pipeline properties, primarily error handling.
  2. Add a Directory origin to represent the data to be processed.
  3. Preview source data to determine field-level details needed for the pipeline.
  4. Use a Stream Selector processor to route records paid by credit card to the primary branch and those paid by cash to an different branch. We'll set up a required field to discard records without a payment type.
  5. Configure a Jython Evaluator to perform custom processing that determines the credit card type based on the credit card number.
  6. Add a Field Masker to mask credit card numbers. Use a required field to discard records without credit card numbers.
  7. Use an Expression Evaluator to add corresponding fields to the cash records so all records have the same data format in the destination. Use data preview to verify the fields to add.
  8. Merge both branches to a Hadoop FS destination.
  9. Add a data rule to raise an alert if too many credit card payments don't have a corresponding credit card number.
  10. Start the pipeline and monitor the results.

Creating a Pipeline and Defining Pipeline Properties

Generally, you can use defaults for pipeline properties, but you should always define how you want to handle error records.

The tutorial writes the records to a local file, but if you prefer to write error records to Kafka, feel free.

  1. Click the Error Records tab, for the Error Records property, select Write to File.
  2. Click the Error Records - Write to File tab and configure the following properties:
    Write to File Property Description
    Directory Enter the directory local to the Data Collector where you want to write error record files.
    Note: The directory must exist to prevent validation errors.
    Files Prefix This defines a prefix for error record files. Delete the default and use "Err_".
    Max File Size When writing to a local directory, you might want to reduce the files size to something more managable, such as 5 MB.

Configure the Origin

The origin represents the source data for the pipeline. When you configure the origin, you define how to connect to the data, and the type of data to be processed.

To process data in a static file, we'll use the Directory origin.

  1. In the Stage list, click the Directory origin. Or, from the Pipeline Creation Help Bar, click Select Origin > Directory.
    The origin displays in the canvas and the Properties panel displays the stage properties.
  2. In the Properties panel, click the Files tab and configure the following properties - use defaults for all other properties:
    Directory Property Value
    Data Format Delimited.
    Files Directory Enter the local directory where you saved the source file.
    File Name Pattern If the file is the only file in the directory, you can use the asterisk wild card (*) or something similarly generic, like *.csv.
  3. For delimited data, additional properties are available, so click the Delimited tab.
  4. The sample data includes a header row, so set the Header Line property to With Header Line.

Route Data with the Stream Selector

To route records that use credit card for payment to a separate branch for processing, we use the Stream Selector processor. We'll also define payment_type as a required field to drop records with no payment type.

Required fields must include data for the record to enter the stage. Records that don't meet the requirement are sent to the pipeline for error handling. For this tutorial pipeline, this means the records will be written to file.

The Stream Selector uses user-defined conditions to route data. Any data not captured by a user-defined condition routes to the default stream.

In this case, because the data is delimited, we use a delimited data record function. The record:dValue function returns the value associated with the specified header so you don't need to know the exact location of the header in the record.

To capture records with credit card payments, use the following condition: 
${record:dValue('payment_type') == 'CRD'}
  1. In the stage library, select the Stream Selector processor and connect it to the pipeline. Or, from the Pipeline Creation Help Bar, click Select Processor to Connect > Stream Selector.
  2. On the General tab, click in the Required Fields property.
  3. To discard records with no payment type information, select the following field: [3]/value.
    The list of fields displays because you've already performed data preview or because the pipeline is valid for data preview. If a list does not appear, you can also enter the field path: [3]/value.
  4. With the Stream Selector selected, in the Properties panel, click the Conditions tab.
    A default condition for the default stream displays. The default stream represents any records not captured by other conditions.
  5. Click the Add icon.
    A condition text box displays, as well as the corresponding output location.
  6. Enter the following condition to capture records where a credit card is used for payment: 
    ${record:dValue('payment_type') == 'CRD'}
    You can use single quotation marks or regular quotation marks around the string values.
    All other records are captured by the default condition and stream.

Use Jython for Card Typing

Next, we'll evaluate credit card numbers and determine the credit card type associated with each number. You can use an Expression Evaluator to do the same calculations, but using a Jython Evaluator is a bit simpler.

The Data Collector provides the JavaScript Evaluator and the Jython Evaluator so you can use custom script to perform processing that is not easily performed using other Data Collector processors.

The Jython script that we're using creates an additional field, credit_card_type, and generates the credit card type based on the first few digits of the credit card number. The script passes an error message if the record has a credit card payment type, but no credit card number.
  1. From the stage library, select the Jython Evaluator processor and connect it to the first output location of the Stream Selector. Or, from the Pipeline Creation Help Bar, click Select Processor to Connect > Jython Evaluator.
    This routes records paid by credit card to the Jython Evaluator.
  2. With the Jython Evaluator selected, in the Properties panel, click the Jython tab.
  3. Use the default record processing mode to process data in batches instead of record by record.
  4. In the Script text box, delete everything below the first set of comments, and then paste the following script: 
    try: 
  for record in records:
    cc = record.value[20]['value']
    if cc == '':
      err.write(record, "Payment type was CRD, but credit card was null")
      continue

    cc_type = ''
    if cc.startswith('4'):
      cc_type = 'Visa'
    elif cc.startswith(('51','52','53','54','55')):
      cc_type = 'MasterCard'
    elif cc.startswith(('34','37')):
      cc_type = 'AMEX'
    elif cc.startswith(('300','301','302','303','304','305','36','38')):
      cc_type = 'Diners Club'
    elif cc.startswith(('6011','65')):
      cc_type = 'Discover'
    elif cc.startswith(('2131','1800','35')):
      cc_type = 'JCB'
    else:
      cc_type = 'Other'

    record.value.append({'header':'credit_card_type','value':cc_type})
    out.write(record)
except Exception as e:
  err.write(record, e.message)
    Note: Incorrect indentation can cause Jython validation errors. For best results, copy the script from the online help. Copying the script from the User Guide PDF can cause incorrect indentation.

Mask Credit Card Numbers

Now that the credit card type has been evaluated, we'll use a Field Masker to mask the credit card numbers.

The Field Masker provides fixed and variable-length masks that mask all data. You can create a custom mask that contains masked positions and revealed positions. And you can use a regular expression mask that defines the string and optionally allows you to reveal one or more groups in the string.

For our pipeline, let's mask all but the last four digits of the credit card number with the following regular expression:
(.*)([0-9]{4})

The expression deliberately defines two groups so we can reveal the second group.

  1. Add a Field Masker processor to the canvas and connect it to the Jython Evaluator.
  2. In the Properties panel, click the Mask tab.
  3. Click in the Fields to Mask field. Scroll down the list of fields to select the field that represents credit card data: [20]/value.
    A list of fields displays when the pipeline is valid for data preview. If the list does not display, you can type in the field path: [20]/value.
  4. To use a regular expression as a mask and display the last 4 digits of the credit card number, configure the rest of the properties as follows:
    Field Masker Property Configuration
    Mask Type Regular Expression
    Regular Expression (.*)([0-9]{4})
    Groups to Reveal 2

Write to the Destination

You can use the Hadoop FS destination to write tutorial data to HDFS or to a local directory. We'll write to the local directory that you set up.

When you configure the output file properties, you define the directory template. This determines the naming convention for the output directories that are created. Use the directory where you want the output files to be written.

  1. Add a Hadoop FS destination to the canvas and and connect it to the Field Masker.
  2. In the Properties panel, click the Hadoop FS tab. If you have a real Hadoop FS installation, you can configure the stage to write to it.
    Or, to use Hadoop FS to write to a local file system, enter the following configuration for the Hadoop FS URI property: file:///.
  3. Click the Output Files tab and configure the following properties. Use the defaults for the rest:
    Hadoop FS Property Configuration
    Data Format Delimited
    Files Prefix By default, Hadoop FS uses the Data Collector ID. That can be useful, but let's simplify and use "Tutorial_" instead.
    Directory Template Use the output directory where you want the files to be written and optionally include constants to indicate the frequency at which the files should be rolled over.

    For example the following directory rolls the output file once a year, but works for this tutorial: <output directory>/${YYYY()}.
    Max Records in File 250000
    Max File Size (MB) If writing to a local file system, you might prefer to lower the file size to something manageable, like 5 or 1.
  4. Click the Delimited tab and configure the following property. Use the defaults for the rest:
    Delimited Property Value
    Header Line With Header Line
That completes the primary branch. Now, let's go back to the Stream Selector and complete the secondary branch.

Add Corresponding Fields with the Expression Evaluator

Use an Expression Editor to add the credit card type fields to the non-credit card stream so all records have the same format when written to the destination.

To do this, let's use data preview to verify how we added the credit card type to the records.

  1. Click the Preview icon.
  2. In the pipeline, click the Jython Evaluator to view the output of that processor.
  3. Expand the first output record, scroll down, and expand index 21. Note the header and value information, and then click Close Preview.

    From this, we can see that we need to add a map with an index of 21, as well as the header and value fields.

  4. Add an Expression Evaluator processor to the canvas and connect it to default output of the Stream Selector.
  5. Click the Expressions tab.
  6. Click the Add icon twice and configure the properties as follows:
    Output Field Expression
    [21] ${emptyMap()}
    [21]/header credit_card_type
    [21]/value N/A
    These fields create a map for index 21, a header field with the name credit_card_type, and a value field that indicates the information is not applicable.
  7. To write the data from this branch to the destination and merge the non-credit card data with the credit card data, link the Expression Evaluator to the Hadoop FS destination.

Create a Data Rule and Alert

The basic pipeline is basically complete. But let's create a data rule and alert. Data rules are user defined rules used to assess data moving through a stream or link between two stages.

The script in the Jython Evaluator sends records to error that were paid by credit card but don't have a credit card number associated with the record. But we can create a data rule and alert to let us know when the count reaches a specified threshold.

  1. Select the link between the Stream Selector and the Jython Evaluator.
    The Data Rules tab displays in the Preview panel.
  2. Click Add.
  3. In the Data Rule dialog box, configure the following properties:
    Data Rule Property Description
    Label Missing Card Numbers
    Condition ${record:value("[20]/value") == ""}
    Sampling Percentage 20
    Alert Text Over 10 missing credit card numbers!
    Threshold Value 10
    With a larger data set, a smaller sampling percentage might be appropriate, but for the purposes of this tutorial, we'll use 20.
  4. Click Save.
    The rule displays in the data rule list.
  5. To enable the data rule and alert, click Active.

Run the Basic Pipeline

Now that the basic pipeline is complete, you can start it by clicking the Start icon.

The console enters Monitor mode, showing summary statistics in the Monitor panel. Odds are good the data alert has triggered, displaying a pop up message and a notification:

You can close the pop up message and explore the information available in the Monitor panel.

Note that when you select an unused part of the canvas, the Monitor panel displays monitoring information for the pipeline. When you select a stage, it displays information for the stage.

When you select the Jython Evaluator, statistics for the stage displays. You can click the Error tab to view the error records.

You can also select the red Alert icon to view information about the data alert and view the sampled error records.

To continue on to the extended tutorial, stop the pipeline.

If you like, you can use data preview to step through the pipeline to review how each stage processes the data.
